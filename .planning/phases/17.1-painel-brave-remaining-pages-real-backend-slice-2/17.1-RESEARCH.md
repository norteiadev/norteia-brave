## Summary

This slice adds two NET-NEW backend features (Duplicados dedup-pairs, Varreduras runs-history) plus
new full-pipeline Kanban stage transitions, then cross-wires already-built views into the finished
`/painel` shell. The riskiest finding is **not** architectural ŌĆö it is that **`compute_embedding`
is still a Phase-1 zero stub** (`brave/core/rio/dedup.py:102-117`): every `RioRecord.embedding`
written by the live pipeline (`brave/core/rio/routing.py:182`) is a 1536-dim zero vector, and
`_cosine_similarity` returns `0.0` for zero vectors (`dedup.py:136`). **There is no real semantic
similarity signal in the data today.** Any "Duplicados pairs ranked by pgvector cosine" feature that
trusts the embedding column will rank everything at 0.0. This reshapes recommendation #1 below.

The existing transition surface is well-built and audited: every mutation routes through
`write_audit()` (`brave/observability/audit.py:22`) and `require_steward_or_bearer`. The Kanban's
`mapDrop` allow-list (`dashboard/lib/painel-actions.ts:54`) is the single client security boundary ŌĆö
extend it, never bypass it. The dashboard already proxies everything through one BFF catch-all
(`dashboard/app/api/[...path]/route.ts`) with server-side Bearer injection, and tests mock at the
network layer with MSW (`dashboard/mocks/`), so the two new endpoints need only new typed clients +
new MSW handler modules ŌĆö no new BFF files.

**Primary recommendation:** Build Duplicados as **compute-on-read** (no materialized table, no new
write path) with similarity computed **in Python from normalized fields** (NOT via the degenerate
pgvector operator), so it is meaningful today and unit-testable offline. Build Varreduras as a thin
`runs_history` envelope table written at engine-start and finalized at orchestrator-idle, with
synced/failed computed on-read from the medallion tables over the run's time window. Add ONE generic
audited stage-transition endpoint per entity for the new backward/force moves rather than a matrix
of per-pair PATCHes. Reuse the existing Pact pattern only conceptually ŌĆö the existing Pact is
braveŌåÆnorteia-api (Python consumer); there is **no** dashboardŌåöcollector Pact and adding one is a
disproportionate new toolchain (see #3).

## Finding 1 ŌĆö Duplicados dedup-pairs backend

### The decisive fact: embeddings are a zero stub today

- `compute_embedding(text)` returns `[0.0] * 1536` ŌĆö explicit Phase-1 stub
  (`brave/core/rio/dedup.py:102-117`).
- It is the ONLY embedding writer: `brave/core/rio/routing.py:182` calls `compute_embedding`, then
  writes it to the RioRecord at lines 192/206. No real embedding client exists anywhere in `brave/`
  (verified: no `embeddings.create` / `text-embedding` / `EmbeddingClient` in the tree).
- `_cosine_similarity` returns `0.0` when either vector norm is 0 (`dedup.py:136-138`), and the
  pgvector `cosine_distance` ordering over all-zero vectors is meaningless.

**Consequence:** the live two-stage dedup (`find_duplicate`, `dedup.py:24-99`) effectively works
**only via Stage 1 (exact `content_hash` match, territorially scoped)**. Stage 2 (vector fuzzy) never
crosses the 0.95 threshold because every similarity is 0.0. A Duplicados view that advertises
"pgvector cosine similarity" will display 0.0 for every pair. **This must be surfaced to the user as
an `[ASSUMED]`/decision point** (see Assumptions Log A1).

### Recommendation: compute-on-read, similarity in Python (LOWER RISK)

Pick **compute-on-read** over a materialized `dedup_pairs` table:

| Option | Risk | Offline-testability |
|--------|------|---------------------|
| **(a) compute-on-read** Ō£ģ | Pure SELECT + Python diff; no new write path; no migration; no staleness | Unit-testable with a `MagicMock` session (mirrors `tests/unit/test_dedup.py`) ŌĆö IF similarity is computed in Python, not SQL |
| (b) materialized `dedup_pairs` table | New migration + a write path wired into the Rio pipeline + backfill + staleness/repair | Needs integration DB to test the write path; more surface |

**Why Python-side similarity, not the pgvector operator:** the moment a read query calls
`RioRecord.embedding.cosine_distance(...)` it requires a real Postgres+pgvector and becomes
`@pytest.mark.integration` (skips silently without `BRAVE_DB_URL` ŌĆö see MEMORY note). Computing the
displayed similarity in Python from `rio.normalized` vs `mar.canonical` fields keeps the whole
Duplicados endpoint unit-testable offline AND makes it meaningful today (unlike the zero embeddings).
When real embeddings eventually land, the same endpoint can add a vector-ranked candidate fetch
behind an integration test without changing the response contract.

### Candidate selection query (territorial-key blocked ŌĆö reuse the dedup invariant)

Mirror `find_duplicate`'s blocking exactly (NEVER compare across UF ŌĆö `dedup.py:39`, `models.py:343`
`ix_rio_territorial_key`). For each non-published candidate, find the published Mar record sharing
its territorial key:

```python
# Source pattern: brave/core/rio/dedup.py:82-91 (same blocking), models.py:176 MarRecord
# Candidates = rio rows NOT yet in Mar (routing in_progress/dlq) that collide with an active Mar row
# on (uf, municipio_id, entity_type). Pair them and diff fields in Python.
from sqlalchemy import select
from brave.core.models import RioRecord, MarRecord

# 1) pull candidate rio rows in scope (optionally filtered by uf)
candidates = session.scalars(
    select(RioRecord).where(
        RioRecord.routing.in_(("in_progress", "dlq")),
        RioRecord.municipio_id.isnot(None),

    ).limit(limit)
).all()

# 2) for each candidate, find the active Mar row(s) on the SAME territorial key
#    (Mar has no municipio_id column ŌĆö join via its source rio_id, or match on
#     canonical['uf']/['municipio']; simplest: go through MarRecord.rio ŌåÆ RioRecord)
#    Recommended: SELECT MarRecord JOIN RioRecord(rio_id) on matching uf+municipio_id+entity_type
```

Note `MarRecord` carries `entity_type`, `source_ref`, `canonical`, `superseded_by_id`
(`models.py:201-223`) but NOT `uf`/`municipio_id` directly ŌĆö those live on its backing `RioRecord`
(`MarRecord.rio` relationship, `models.py:229`) and inside `canonical`. The pairing join goes
`MarRecord ŌåÆ rio_id ŌåÆ RioRecord` and blocks on the RioRecord territorial key. Keep the active-row
filter `MarRecord.superseded_by_id.is_(None)` (`models.py:191`) so superseded history isn't paired.

### matched / diverged field labels

Compute in Python by comparing the candidate's `rio.normalized` (`models.py:140`) against the Mar
row's `canonical` (`models.py:207`), over the canonical field allow-list (the same keys
`promote_to_mar` keeps ŌĆö it strips the `*_value` score inputs, `mar/service.py:46-50`). For each
shared key: `matched` if equal, `diverged` if both present and unequal. Cheap, deterministic, fully
offline-testable.

|--------|-------------------------|-------|--------------|
| **merge** (merge candidate into existing Mar) | Treat the candidate as the newer truth: set `candidate.routing="mar"` and call `promote_to_mar(session, candidate_rio)` ŌĆö its supersession path (`mar/service.py:71-101`) appends a new active Mar row and points the old one's `superseded_by_id`, idempotent by `source_ref`. ## Finding 2 ŌĆö Varreduras runs-history backend

### Where runs live today: Redis only, no DB trail

Engine state is a handful of Redis keys, NOT a DB row (`brave/core/engine.py:42-52`):
`brave:engine:state|current_uf|ufs_done|ufs_total|depth|source`. `get_status` (`engine.py:149-158`)
serializes them. The orchestrator `engine_sweep_run` (`brave/tasks/pipeline.py:1864`) fans out
per-UF producer tasks and tracks a local `dispatched` counter (`pipeline.py:1911`) but **does NOT
await downstream results** ŌĆö the actual Mar sync happens later in separate `push_*` tasks. So
**synced/failed are not naturally available at run-completion time**; only "UFs dispatched" is.

### Recommended schema: `runs_history` envelope + on-read aggregation

```python
# brave/core/models.py ŌĆö new model (mirrors existing table style)
class RunHistory(Base):
    __tablename__ = "runs_history"
    id: Mapped[uuid.UUID] = mapped_column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    started_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), nullable=False,
                                                 server_default=func.now(), index=True)
    ended_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)
    ufs: Mapped[dict] = mapped_column(JSON, nullable=False)         # ["BA", ...] (run can be multi-UF)
    source: Mapped[str] = mapped_column(String(32), nullable=False) # default | tripadvisor
    depth: Mapped[str] = mapped_column(String(32), nullable=False)  # nascente|nascente_rio|nascente_rio_mar
    lane: Mapped[str] = mapped_column(String(32), nullable=False, server_default="both")
    ufs_total: Mapped[int] = mapped_column(Integer, nullable=False, default=0)
    ufs_dispatched: Mapped[int] = mapped_column(Integer, nullable=False, default=0)
    # nullable aggregate snapshot, filled at finalize; the LIST endpoint may recompute on-read
    total: Mapped[int | None] = mapped_column(Integer, nullable=True)
    synced: Mapped[int | None] = mapped_column(Integer, nullable=True)
    failed: Mapped[int | None] = mapped_column(Integer, nullable=True)
    status: Mapped[str] = mapped_column(String(16), nullable=False, server_default="running")
    # running | concluido | parcial | falha
```

Design-canvas columns (CONTEXT `seedRuns`: In├Łcio/UF/Fonte/Profundidade/Total/Sincr./Falhas/Status)
all map onto the above. Single-UF display: render one row per UF if the operator started one UF, or
show the `ufs` array. The design seed uses one row per (uf, started_at) ŌĆö keep it simple: a row may
represent a single-UF start or a multi-UF sweep; the list returns whatever was started.

### Write points (exact)

1. **On start ŌĆö `brave/api/routers/engine.py:engine_start`** (the authenticated edge, `engine.py:87`).
   Currently it has only `redis` injected. **Add `db: Session = Depends(get_db)`** and INSERT a
   `RunHistory` row (status="running", ufs, source, depth, lane, ufs_total=len(ufs)) AFTER the depth
   (`engine.py:108`) and source (`engine.py:117`) 422 guards pass and AFTER `start_run` succeeds
   (`engine.py:123`, so a 409 "already running" does not create a row). Capture `run_id`, persist it
   to Redis (`brave:engine:run_id`) so the orchestrator + status endpoint can find it, and pass
   `run_id` into `engine_sweep_run.delay(...)` (`engine.py:135`).
2. **On finalize ŌĆö `engine_sweep_run`** (`pipeline.py:1864`). The task already opens its own
   `redis_lib` client (`pipeline.py:1902`); give it a DB session (the pipeline module already creates
   sessions elsewhere). When the UF loop exits and the engine returns to idle, UPDATE the row:
   `ended_at=now()`, `ufs_dispatched=dispatched` (`pipeline.py:1911`), `status="concluido"` (or
   `"parcial"` if Stop was requested mid-run ŌĆö detectable via the same state read the loop already
   does). Wrap in try/except so a runs-history write failure NEVER aborts the sweep (mirror the
   best-effort audit posture in `tripadvisor_session.py:292-335`).
3. **synced/failed:** do NOT try to thread these through async producer tasks. **Compute on-read** in
   the LIST endpoint over `[started_at, ended_at]` for the run's UFs:
   - `synced` = count of `MarRecord` with `published_at` in window (`models.py:224`).
   - `failed` = count of `RioRecord` routed `dlq`/`descarte` with `processed_at` in window
     (`models.py:147`), plus `PoisonQuarantine` rows in window (`models.py:310`) for true engine
     failures. Document this as an approximation (a record from a prior run re-touched in-window
     could be counted) ŌĆö acceptable for an ops dashboard; flag as A4.

### Recommendation: ONE generic, audited, guarded transition endpoint per entity

Adding a per-(source,target) PATCH matrix would be ~10 endpoints with near-identical guard/audit
boilerplate. Instead add ONE generic transition endpoint per entity that takes the target column and
internally maps to a SMALL, explicit allow-list (the server-side twin of `mapDrop`):

```jsonc
// PATCH /api/v1/destinos/{rio_id}/transition   (require_steward_or_bearer)
// PATCH /api/v1/atrativos/{rio_id}/transition  (require_steward_or_bearer)
// body: { "to": "nascente|rio|whatsapp|mar|dlq|descarte", "expected": "<current col>" }
//   - server validates the (current_state ŌåÆ to) edge against an explicit allow-list
//   - illegal/unsupported edge ŌåÆ 409 "transi├¦├Żo n├Żo suportada" (client reverts + toasts)
//   - legal edge ŌåÆ performs the mutation, writes audit, returns the new column
```

Why this shape:
- **Single audit/guard choke point** ŌĆö every transition flows through `write_audit` + the same
  `require_steward_or_bearer` + an `expected`-state optimistic-concurrency check (reuse the
  `advance_sub_state` 409 idea, `cms.py:639`). This is the auditable, safe surface the CONTEXT wants.
- **The allow-list is the security boundary**, server-side. It encodes exactly which backward/force
  edges are permitted (e.g. `dlq ŌåÆ rio` re-open via reprocess; `rio ŌåÆ descarte`). Edges that are
  unsafe (anything OUT of `mar`, anything INTO `nascente` since nascente has no draggable rows) stay
  rejected with 409 ŌĆö never silently performed.
- **Client `mapDrop` mirrors it** (`painel-actions.ts:54`): extend `mapDrop` to emit a
  `{kind:"transition", to, expected}` action for the new edges and call the new endpoint; keep
  returning `null` (revert + toast) for everything not in the server allow-list. The two allow-lists
  must agree ŌĆö document them as a paired contract.

Concretely, the NEW edges this slice should support (everything else stays a `null`/409 no-op):
- Atrativo reprocess (`dlq ŌåÆ rio`): currently missing ŌĆö add it (atrativos can re-enter Rio).
- Force-to-`dlq` for either entity from `in_progress` (manual "send to review"): cheap, audited.
- Backward `mar ŌåÆ *`: **NOT supported** (depublish deferred) ŌĆö keep 409.
- Into-`whatsapp` for an atrativo only valid from `aguardando_consulta_whatsapp` ŌĆö delegate to the
  existing gate approve (`atrativos_gate.py:318`), do not duplicate its outreach dispatch.

## Common Pitfalls

### Pitfall 1: Trusting pgvector similarity that is actually zero
**What goes wrong:** Duplicados ranks/filters pairs by `embedding.cosine_distance` and every pair
scores identically (0.0), so the view is noise.
**Why:** `compute_embedding` is a Phase-1 stub (`dedup.py:102-117`); real embeddings were deferred.
**Avoid:** compute the displayed similarity in Python from `normalized`/`canonical` fields; treat the
embedding-derived number as a labeled placeholder (`similarity_source`). Surface the limitation to the
user (A1).
**Warning sign:** a unit test asserting a specific non-zero cosine value ŌĆö it can only pass against
fabricated non-zero embeddings, not real pipeline data.

### Pitfall 2: A pgvector operator in the default (offline) test path
**What goes wrong:** `RioRecord.embedding.cosine_distance(...)` in a read query forces a real
Postgres+pgvector; the test becomes integration and **skips silently** without `BRAVE_DB_URL`
(MEMORY note), masking regressions (green worktree / red main).
**Avoid:** keep Duplicados selection blockable + diffable in Python; reserve any pgvector SQL for
explicit `@pytest.mark.integration` tests.

### Pitfall 3: Writing a runs_history row before the 409/422 guards
**What goes wrong:** inserting the run row at the top of `engine_start` records phantom runs for
rejected starts (bad depth/source, or "already running").
**Avoid:** insert ONLY after the depth (`engine.py:108`) + source (`engine.py:117`) guards pass AND
`start_run` returns True (`engine.py:123`).

### Pitfall 4: Merge creating a 2nd active Mar instead of merging
**What goes wrong:** `promote_to_mar` keys supersession on `source_ref` (`mar/service.py:41,64`). If
the candidate and the existing Mar have different `source_ref`, "merge" appends a 2nd active Mar
(two canonical rows for one place) instead of superseding.
**Avoid:** v1 merge supersedes only when source_refs match; otherwise 409 "fontes distintas" (A2).

### Pitfall 5: Descarte/backward drop out of the `mar` column
**What goes wrong:** a backward drag from Sincronizado looks like it should "un-publish," but plain
descarte is 409-blocked for already-Mar records (`cms.py:347-362`) and there is no depublish flow.
**Avoid:** map any `mar ŌåÆ *` drop to a client-side revert + toast; never add a new endpoint that
silently mutates Rio while leaving the canonical Mar record live downstream.

### Pitfall 6: Double-prefix BFF path in new MSW handlers
**What goes wrong:** handler registered at `/api/v1/...` never intercepts because the browser path is
`/api/api/v1/...` (`engine.ts:15`, "Pitfall 5").
**Avoid:** register new dedup/runs handlers at `http://localhost:3000/api/api/v1/...`.

@router.patch("/api/v1/dedup/pairs/{candidate_rio_id}/resolve",
              dependencies=[Depends(require_steward_or_bearer)])
def resolve_pair(candidate_rio_id: uuid.UUID, body: ResolveBody, db: Session = Depends(get_db)):
    rio = db.get(RioRecord, candidate_rio_id)
    if rio is None: raise HTTPException(404, "RioRecord not found")

    if body.action == "discard":

                record_id=rio.id, before_state=before,
                after_state={"routing": rio.routing}, actor="steward")
    db.commit()
    return {"status": "ok", "action": body.action}
