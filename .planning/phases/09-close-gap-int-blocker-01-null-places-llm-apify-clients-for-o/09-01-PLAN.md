<objective>
Close INT-BLOCKER-01: production Celery tasks import `tests.fakes.*` in their
default (`run_real_externals=False`) branch, but the production wheel ships only
`packages=["brave"]` (pyproject.toml:62-63). Under the documented default config
every sweep/discovery/FSM task raises `ModuleNotFoundError: No module named
'tests'`. It is masked today only because the repo `.env` forces
`RUN_REAL_EXTERNALS=true`.

This plan adds three in-package, production-safe Null clients under
`brave/clients/`, rewires the 8 offline-branch import sites in
`brave/tasks/pipeline.py` to use them, and adds a durable regression guard
proving the `brave` package never imports the `tests` tree.

This is a bounded, behavior-preserving wiring fix — NOT a feature phase. No
real-client behavior, score engine, FSM logic, or other audit WARNINGs change.

Purpose: Make the default offline task branch importable from the production
wheel so destinos sweeps and atrativos FSM auto-advance run without crashing.
Output: 3 new Null client modules, rewired pipeline.py, 1 regression-guard test.
</objective>

GREP HYGIENE (critical): existing `brave/` files (base.py, __init__.py,
whatsapp.py, null_*.py) contain the substring `tests/fakes` and `tests.fakes`
inside docstrings/comments. The new Null modules will also reference "tests/" in
their docstrings (mirroring null_mtur.py). So the regression guard MUST match
import STATEMENTS only — lines matching `^\s*(from|import)\s+tests\b` — not the
bare substring `tests.fakes`. A bare substring check would false-positive on
docstrings and would not be a valid gate.

Env var note: `brave/config/settings.py` uses `env_prefix=""`, so the setting
`run_real_externals` is controlled by the env var `RUN_REAL_EXTERNALS`. The repo
`.env` forces `RUN_REAL_EXTERNALS=true`, which masks this bug. All verification
that exercises the offline branch MUST run with `RUN_REAL_EXTERNALS` unset/false.
