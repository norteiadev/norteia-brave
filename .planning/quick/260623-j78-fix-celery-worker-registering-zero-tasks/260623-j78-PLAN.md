# Quick Task 260623-j78: Fix Celery worker registering zero tasks

## Problem
`brave/tasks/celery_app.py` called `app.autodiscover_tasks(["brave.tasks"])`. Celery's
`autodiscover_tasks` defaults `related_name="tasks"`, so it tried to import
`brave.tasks.tasks` — a module that does not exist. The real `@shared_task`
definitions live in `brave/tasks/pipeline.py`, so the worker booted with an empty
`[tasks]` list and silently dropped every dispatched task. Surfaced live during the
Phase 10 "Apenas nascente" test: an engine sweep hung in `running` with `nascente=0`;
adding `--include brave.tasks.pipeline` made it run.

## Tasks
1. `brave/tasks/celery_app.py` — pass `related_name="pipeline"` to `autodiscover_tasks`
   so the worker imports the real task module at boot (matches the existing comment intent).
2. `tests/unit/test_celery_task_registration.py` (new) — offline regression guard:
   after `app.loader.import_default_modules()`, assert `brave.engine_sweep_run`,
   `brave.sweep_uf`, `brave.process_nascente`, `brave.push_mar` are all in `app.tasks`
   (no manual `--include`).

## Verify
- `.venv/bin/python -m pytest tests/unit/test_celery_task_registration.py -q` (offline)
- Live: `celery -A brave.tasks.celery_app:app worker` (no `--include`) shows the 4 tasks in `[tasks]`.

## Out of scope
End-to-end sweep verification (Phase 10 live test already proved it).
