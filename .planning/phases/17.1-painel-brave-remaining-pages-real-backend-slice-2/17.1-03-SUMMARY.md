# Phase 17.1 Plan 03: Generic audited per-entity stage-transition endpoint Summary

**One generic `transition` endpoint per entity (destino, atrativo) gated by a server-side `(expected, to)` edge allow-list — the server twin of the client mapDrop — that 409-blocks every mar→* and any unmapped edge, audits each legal move, and reuses existing pipeline helpers (no new machinery).**

## Performance

- **Duration:** ~25 min
- **Completed:** 2026-06-28
- **Tasks:** 2
- **Files modified:** 3 (2 routers + 1 new test module)

## Accomplishments
- `transition_destino` (cms.py): `_ALLOWED_EDGES` allow-list — rio→mar/descarte/dlq, dlq→rio/mar/descarte; reuses `validate_and_promote_rio` (→mar) and `reprocess_record` (dlq→rio reopen); flips routing for descarte/send-to-review.
- `transition_atrativo` (atrativos.py): `_ATRATIVO_ALLOWED_EDGES` — rio→dlq/mar/descarte, dlq→rio reopen (NEW backward edge), and the into-whatsapp edge that delegates to the audited `approve_whatsapp_gate` (only from `aguardando_consulta_whatsapp`).
- `TransitionBody` (`extra="forbid"`) + `_ROUTING_TO_COLUMN` optimistic-concurrency guard, shared across both endpoints (atrativos imports them from cms — the documented paired contract).
- mar→* and every unmapped edge return 409 `transição não suportada` BEFORE any mutation; every legal edge writes a `transition_<to>` audit row then commits.
- `tests/unit/test_transitions.py`: 20 fully-offline tests (MagicMock session) — exhaustive edge tables for both entities + endpoint guards (mar→* never mutates, concurrency 409, 404, whatsapp delegate only from aguardando).

## Decisions Made
- Kept two distinct allow-lists rather than one shared table — destino and atrativo have different legal edges (e.g. atrativo has the whatsapp-delegate edge and a mar-ready promote-override; destino promotes via DLQ validate).
- For the atrativo into-whatsapp edge, fully delegate to `approve_whatsapp_gate` and return its result (no second audit/commit) so the outreach dispatch is never duplicated; added an explicit `aguardando_consulta_whatsapp` guard in the transition endpoint as well so the 409 is enforced even when the delegate is mocked.
- `reprocess_record` (dlq→rio) re-scores via §7.6, so the record may re-route; the audit `after_state` records the actual resulting routing alongside the requested `to` column.
