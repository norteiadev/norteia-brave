# Phase 17.1 Plan 01: Duplicados backend Summary

**Compute-on-read dedup candidate↔Mar pairs (territorial-key blocked, Python field similarity) + an audited merge/keep/discard resolve endpoint that unions provenance into the existing Mar — no new Mar row, no 409, no pgvector operator.**

- **Completed:** 2026-06-28

## Accomplishments
- `GET /api/v1/dedup/pairs` (require_bearer): pairs each pending candidate Rio (routing in_progress|dlq, has municipio_id) with the ACTIVE Mar on its territorial key; matched/diverged field labels + a labeled similarity computed in Python from `normalized` vs `canonical` over the canonical allow-list. No `cosine_distance` / pgvector operator in the read path.
- `PATCH /api/v1/dedup/pairs/{candidate_rio_id}/resolve` (require_steward_or_bearer): merge / keep / discard, with `write_audit` on every action.
- **merge (LOCKED A2):** unions the candidate's source_ref into the existing Mar's `provenance["merged_source_refs"]` and routes the candidate Rio → `descarte`. No new MarRecord, `mar.source_ref` never mutated (uq_mar_active_source_ref stays satisfied), no 409, no promotion/supersession path.
- 12 offline unit tests (MagicMock session, RUN_REAL_EXTERNALS unset) — field diff, token similarity, territorial-key blocking, exact envelope shape, merge provenance union, discard/keep dispatch, and 404 paths.

## Decisions Made
- **Candidate source_ref derivation:** RioRecord has no `source_ref` column; derived as `rio.canonical_key or str(rio.id)`, the same convention `promote_to_mar` uses (`mar/service.py:41`). The plan text literally referenced `rio.source_ref`, which would raise `AttributeError`.
- **similarity_source = "embedding_stub":** followed the plan's explicit action value; the numeric `similarity` is Jaccard token overlap of name/municipio/UF — an honest placeholder for the deferred real-embedding cosine (A1).
- **merge semantics:** implemented per LOCKED CONTEXT A2, which the plan flags as overriding the stale RESEARCH Pitfall 4 (the old "supersede or 409" approach).
