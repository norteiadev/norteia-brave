<objective>
Add the on-demand ops trigger (ORCH-03): a `brave.cli sweep` subcommand (the
required surface) plus an optional Bearer-guarded `POST /api/v1/sweep` endpoint,
so an operator can kick a UF sweep for destinos and/or atrativos without waiting
for the beat schedule. Both reuse the established dispatch-Celery-then-run-inline
fallback so a sweep works even with no broker. This builds on the now-real
`sweep_uf` (plan 05-01) and the auto-chaining `discover_atrativo_task`
(plan 05-02). Closes ORCH-03 and the ops-trigger half of ORCH-04.

Purpose: give operators a manual lever over the same automation the beat drives,
without a worker dependency.
Output: CLI `sweep` command + optional `/api/v1/sweep` router + offline tests.
</objective>

| ops trigger → sweep_uf / discover_atrativo_task | reuses the same producer/chain tasks; no new privilege |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-05-08 | Denial of Service | repeated manual sweeps amplifying cost | mitigate | sweep_uf/discover are idempotent (store_raw dedup + sub_state guards); the existing USD cost guard (OBS-02) caps DeepSeek spend; the trigger adds no new cost path |

| T-05-SC | Tampering | npm/pip installs | accept | no new packages added; FastAPI + deps already vendored |
