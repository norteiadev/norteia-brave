<objective>
Fix the Celery queue routing bug where beat-scheduled tasks (3 entries in BRAVE_BEAT_SCHEDULE) are
pinned to the `brave.sweep` queue while every `.delay()` fan-out and the documented worker command
use the default `celery` queue — causing beat tasks to be silently dropped.

Unify on a single default `celery` queue (Solution A, locked by operator). This is a single-host /
all-in-one deployment; dedicated queue lanes (task_routes + separate worker pools) are explicitly
deferred as a future scale-up.

Purpose: Beat-dispatched sweeps (daily UF sweeps, daily atrativo discovery, 10-min TA keepalive)
currently land on `brave.sweep`, which no running worker consumes. This was the confirmed root cause
of SPIKE-1 "ta-keepalive never fired". After this fix, one undifferentiated worker processes all
task types.

Output: Three config edits (beat_schedule.py, celery_app.py, docker-compose.yml), one new
regression test, one doc update (CLAUDE.md Conventions).
</objective>
