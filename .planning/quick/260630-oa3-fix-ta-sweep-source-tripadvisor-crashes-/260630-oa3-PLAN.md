<objective>
Remove the TA-destinos Step 1 from sweep_tripadvisor's per-UF path and rebuild
destino_rio_map from ALL destination RioRecords in the UF (Mtur/IBGE, authoritative)
instead of only tripadvisor-sourced ones.

Purpose: Every UF crashes with "No destinations queryId configured" because the sweep
calls TripAdvisorDestinosIngest.produce() before the atrativos step — but no TA
destinos QID was ever captured (_DESTINATIONS_QID=None, fail-loud by design). Per the
locked architecture decision, destinos = Mtur/IBGE (origem=100); TripAdvisor = atrativos
only. Atrativos' ibge_code lookup already works (ftx: fetch_attraction_geo → cityName →
IBGE). The ONLY broken wiring is that the sweep still runs TA-destinos first and anchors
the map to source=="tripadvisor".

Output: Fixed pipeline.py, updated atrativos.py docstring, updated unit tests.
</objective>
