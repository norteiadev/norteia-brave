<objective>
Make the Atrativos sub-state FSM actually auto-advance. Two gaps are closed:
(1) **finding #1** — today DiscoveryAgent only writes Nascente; no production
code creates the Rio record nor sets `sub_state='discovered'`, so the chain's
`sub_state='discovered'` query has no anchor and the contact_finder precondition
(`sub_state == 'discovered'`) can never be met. (2) the orphaned
`find_contacts_task` / `gather_signals_task` are never enqueued, so the FSM
stalls at discovery. This plan initializes the FSM substrate at discovery and
wires the self-enqueue chain
`discovered → contacts_found → signals_gathered → [borderline] aguardando_consulta_whatsapp → STOP`.
Closes ORCH-02 and the Atrativos half of ORCH-04.

Purpose: deliver the "discovery automatically drives records to the human
WhatsApp gate" promise — and stop there (no auto-send).
Output: discovery-side FSM init + enqueue tails on two tasks + an offline e2e
chain test.
</objective>

  Creates the Rio (routing.py:179) but does NOT set sub_state (defaults to NULL).

  guard requires sub_state=="contacts_found" (signal_agent.py:172). So the Rio must be created AND
  advanced to 'discovered' at discovery time, or the chain has no anchor.

  state_machine.py:28. Setting initial state: advance_sub_state(session, rio, expected_state=None,
  next_state="discovered", actor="discovery_agent"). This is EXACTLY how the existing e2e test seeds
  the FSM (test_atrativos_lane_e2e.py:319-329).

  (contact_finder_agent.py:72 `if rio.sub_state != "discovered": return`;
