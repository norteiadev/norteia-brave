# Phase 4 — UI Design Contract

> Visual and interaction contract for the **norteia-brave Territorial CMS** (internal operations dashboard). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Surface class:** internal ops CMS / territorial CMS — utility-first, information-dense, function over polish, fully accessible. NOT a consumer product. Six surfaces over a FastAPI REST surface, single operator behind Bearer auth.
>
> **Autonomous-mode note:** This ran in the `/gsd:ui-phase` chain with no interactive questions (hard constraint). Every contract value below is **locked from CONTEXT D-03..D-06 + RESEARCH §5 + ROADMAP Phase 4**, or set to the shadcn/Tailwind-v4 default and noted inline as `[default]`. The shadcn registry was NOT initialized (no `npx`/install permitted here); `dashboard/` is greenfield (`.gitkeep` only). `npx shadcn init` runs at scaffold time in the Foundation slice — initialize with the preset recorded below.

## Spacing Scale

Declared values (multiples of 4; maps directly onto Tailwind's 4px-base scale — `p-1`=4 … `p-16`=64):

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `1` | Icon-to-label gaps, inline badge padding, score-bar internal gaps |
| sm | 8px | `2` | Compact element spacing, table cell padding (`px-2`), chip gaps |
| md | 16px | `4` | Default element spacing, card padding, form field gaps |
| lg | 24px | `6` | Section padding, panel header/body separation, master/detail column gap |
| xl | 32px | `8` | Layout gaps between major regions (queue list ↔ detail panel) |
| 2xl | 48px | `12` | Major section breaks, empty-state vertical centering |
| 3xl | 64px | `16` | Page-level top/bottom breathing room (sparingly — this is a dense console) |

Exceptions:
- **Interactive icon-only controls (row actions, header pills) target 44×44px hit area** even when the visual glyph is 16–20px — padding pads to the touch target. (Accessibility minimum; the only deliberate non-pure value, achieved via padding not a raw size token.)
- Table rows are **36px** tall (dense ops requirement) — a layout dimension, not a spacing token; row internal padding stays on the 4px scale (`py-1`/`px-2`).

## Typography

Exactly 4 sizes, exactly 2 weights. Geist Sans for prose/labels; Geist Mono for data (IDs, JSON, scores, tokens, USD).

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (regular) | 1.5 | Table cells, panel field values, transcript text, descriptions. 14px (not 16px) is the deliberate dense-ops body size. |
| Label | 12px | 600 (semibold) | 1.4 | Column headers, score-criterion labels, badge text, form labels, monitor tile captions (often uppercase tracking-wide) |
| Heading | 20px | 600 (semibold) | 1.2 | Panel/section titles ("DLQ Review", "Brave Monitor", "§7.6 Breakdown") |
| Display | 28px | 600 (semibold) | 1.2 | Monitor volume-tile numerals + the prominent per-record total score readout |

- **Weights: exactly two** — regular 400 (all running text/data) and semibold 600 (every emphasis: headings, labels, active nav, score totals). No bold-700, no light-300.
- **Monospace data:** numeric/identifier content (record IDs, `canonical_key`, `score`/`score_breakdown` values, USD cost, model slugs, raw Nascente JSON) renders in Geist Mono at the Body or Label size — tabular-nums for column alignment.

## Color

shadcn/ui neutral base + Tailwind-v4 CSS variables. The 60/30/10 split maps onto shadcn semantic tokens (values below are the shadcn **neutral, dark default** for the primary console theme; light theme uses the inverse shadcn neutral set).

| Role | Value | Token | Usage |
|------|-------|-------|-------|
| Dominant (60%) | `oklch(0.145 0 0)` (near-black neutral) | `--background` / `--card` base | App background, page canvas, table body, panel surfaces |
| Secondary (30%) | `oklch(0.205 0 0)` (raised neutral) | `--card` / `--muted` / `--sidebar` | Sidebar nav, queue-list column, card/tile surfaces, table header, drawer chrome |
| Accent (10%) | `oklch(0.62 0.19 255)` (blue) | `--primary` | See reserved-for list below |
| Destructive | `oklch(0.58 0.22 27)` (red) | `--destructive` | Reject / descarte / opt-out / failure alerts only |

Accent (`--primary` blue) is reserved for — and ONLY for:
- The **primary action button in the active context** (one per view: "Approve" in DLQ detail, "Approve outreach" in the gate, "Sign in" on the login gate).
- The **active/selected state** of the left-nav item and the currently-selected queue row.
- **Focus rings** (`--ring`) on keyboard-focused interactive elements (accessibility).
- The **"Mar" / high-confidence (≥85) routing badge** and the primary series in monitor/funnel charts.
- Never on: body text, table borders, every button, generic links, or decoration.

Semantic status colors (NOT part of the 10% accent budget — these are data-encoding, used only on small badges/dots/score bars, never as large fills):
- **Mar / ≥85 / funcionando / approved** → green `oklch(0.65 0.17 150)`.
- **DLQ / 51–84.9 / borderline / pending** → amber `oklch(0.75 0.16 80)`.
- **descarte / ≤50 / rejected / CLOSED / opted-out** → red (shares `--destructive`).
- **§7.6 score bars** use a single neutral fill with a green/amber/red cap by threshold — they are quantitative, not decorative; keep one hue per bar, label-driven, never a rainbow.

`destructive` reserved for: Reject, Descarte, Opt-out display, batch-reject confirmation, the failure-alert tile/banner (PoisonQuarantine count > 0, RED WhatsApp quality flag). Never for ordinary "Cancel".

## Copywriting Contract

PT-BR is the operator's language (single internal PT-BR ops tool — REQUIREMENTS "Out of Scope: i18n"). UI chrome is PT-BR; domain terms keep their canonical pipeline spelling (Nascente / Rio / Mar / DLQ / descarte / §7.6 criteria). Voice: terse, imperative, operator-to-operator. No marketing tone, no exclamation.

| Element | Copy |
|---------|------|
| Primary CTA — DLQ approve | **"Validar e publicar"** (sets validação-humana=100 → re-score → push if Mar) |
| Primary CTA — DLQ batch | **"Validar lote por estado"** (with selected UF + count: e.g. "Validar lote — BA (24)") |
| Primary CTA — WhatsApp gate | **"Aprovar contato"** |
| Primary CTA — login gate | **"Entrar"** |
| Secondary action — re-score on edit | **"Salvar e reprocessar"** (inline edit → triggers re-score; queue refetches) |
| Secondary action — reprocess | **"Reprocessar"** |
| Empty state heading — DLQ | **"DLQ vazia para este estado"** |
| Empty state body — DLQ | **"Nenhum registro aguardando revisão em {UF}. Selecione outro estado ou aguarde a próxima leva do pipeline."** |
| Empty state heading — gate | **"Fila de gate vazia"** |
| Empty state body — gate | **"Nenhum atrativo em aguardando_consulta_whatsapp. Novos candidatos aparecem conforme o pipeline pontua registros borderline."** |
| Empty state heading — conversations | **"Sem conversas ainda"** |
| Empty state body — conversations | **"Nenhuma conversa de WhatsApp registrada. As transcrições aparecem após a primeira saída aprovada no gate."** |
| Empty state — charts (monitor/cost/funnels) | **"Sem dados no período"** + body **"Ajuste a janela de tempo ou aguarde atividade do pipeline."** |
| Loading state | Skeleton rows/tiles (shadcn `Skeleton`), never a blank flash; tables show 5 skeleton rows, tiles show shimmer numerals. |
| Error state — fetch failure | Heading **"Não foi possível carregar"** · body **"Falha ao consultar a API ({status}). Verifique se o serviço Brave está no ar e tente novamente."** + **"Tentar novamente"** button (refetch). |
| Error state — 401 / auth | Heading **"Sessão expirada ou token inválido"** · body **"Faça login novamente para continuar."** → redirect to login gate. (MSW 401 case — RESEARCH §3.) |
| Mutation success toast | **"Registro validado → Mar"** / **"Registro rejeitado → descarte"** / **"Contato aprovado — saída enfileirada"** (state-transition-explicit, never just "Sucesso"). |
| Destructive confirmation — DLQ reject | **"Rejeitar registro?"** — "Este registro vai para descarte e não será publicado no Mar. Ação reversível só via reprocessamento. **Rejeitar**" |
| Destructive confirmation — gate reject | **"Rejeitar atrativo?"** — "O atrativo não será contatado por WhatsApp e seguirá para DLQ/descarte conforme a pontuação. **Rejeitar**" |
| Destructive confirmation — batch | **"Validar {n} registros de {UF} em lote?"** — "Todos serão marcados como validados pelo humano e reprocessados. **Validar lote**" (batch validate is high-impact → confirm even though non-destructive). |

Destructive actions in this phase: **DLQ reject (→descarte)**, **gate reject**. Both require an explicit shadcn `AlertDialog` confirmation with a red (`destructive`) confirm button and a named action verb (never bare "OK"). Batch validate, while not destructive, also requires confirmation due to blast radius.

PII / LGPD copy rule (RESEARCH §3 R3): phone numbers in gate/conversation views are **masked** (e.g. `+55 11 9••••-••42`); never render full `phone_e164` to the browser. Label masked fields **"telefone (minimizado)"**.

## Registry Safety

| Registry | Blocks Used | Safety Gate |
|----------|-------------|-------------|
| shadcn official | `button`, `input`, `label`, `card`, `table`, `dialog`, `alert-dialog`, `drawer`/`sheet`, `tabs`, `badge`, `skeleton`, `toast`/`sonner`, `select`, `dropdown-menu`, `tooltip`, `separator`, `scroll-area`, `chart` (Recharts wrapper), `sidebar`, `form`, `progress` (score bars), `avatar` (optional) | not required (official registry) |
| third-party | **none** | not applicable |

No third-party / custom registries are declared for this phase. All components come from the shadcn official registry (CONTEXT D-05). The registry-vetting gate is therefore not triggered. If a future slice wants a third-party block, it must re-run `/gsd:ui-phase` and pass the `shadcn view` vetting gate before the block enters this contract.

Bespoke components (composed from official primitives, no new registry):
- **`ScoreBreakdownPanel`** — the signature component: the five §7.6 criteria (origem 30% · completude 20% · corroboração 20% · atualidade 15% · validação-humana 15%) as labeled horizontal score bars (shadcn `Progress` + `Label`), each showing weight, raw value, and weighted contribution, with the total score as the Display-size readout. Threshold cap color (green/amber/red) per bar.
- **`ReviewPanel` + `QueueList`** — the master-detail scaffold reused by DLQ and the WhatsApp gate (CONTEXT D-06): state-filtered list (BA/RJ/SP/SC/CE/PE ordered first) → detail (Nascente payload / Rio normalized / `ScoreBreakdownPanel` / signals / WhatsApp log) → action bar.
- **`StatusBadge`** — routing/sub_state badge using the semantic status colors above.
