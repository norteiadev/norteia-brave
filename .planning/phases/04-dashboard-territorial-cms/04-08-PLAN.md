<objective>
Funnels + Conversations slice (DASH-05) — the final surface, and the one with the open R2 design decision. **Resolve R2 as Option B (RESEARCH §4, recommended):** add a thin append-only `conversation_message` table written by the existing outreach/resume tasks, so the transcript read endpoint is a trivial `SELECT` that stays offline-testable, LGPD-minimizable, and decoupled from LangGraph's internal checkpoint serialization. Then add `GET /api/v1/funnels` (counts by UF/source across pipeline stages) and `GET /api/v1/conversations[/{rio_id}]`, and build the funnels + conversations views.

Purpose: DASH-05 has no backing endpoints (funnels, transcript) and transcripts live un-readably in LangGraph checkpoints (RESEARCH §4) — this slice fills both and ships the views end-to-end. Option B is chosen over Option A because coupling a read endpoint to LangGraph blob serialization is brittle across upgrades and awkward to test offline; an append-only log mirrors the project's "logic+state in our own tables" posture (like `ConsentLog`).
Output: `ConversationMessage` model + migration 0005 + two task write-points + `get_funnels`/`get_conversations` endpoints + funnels/conversations views + offline tests.
</objective>
